WordPress.org Pre-Submission Checklist
Plugin: XPressUI WordPress Bridge
Date: 2026-03-27

But

Cette checklist sert a preparer une soumission wordpress.org avec le maximum de chances d'acceptation rapide.


Bloc A - Bloqueurs probables

[x] Revoir l'architecture des workflow packages
- Point re-verifie le 27 mars 2026.
- Le plugin ne sert plus de page package dans un iframe.
- Le rendu du shell est maintenant inline dans WordPress.
- Le runtime light est embarque dans le plugin.
- Le package reste un artefact importe, mais le risque "iframe / mini-app servie depuis uploads" n'est plus un bloqueur actif.

[ ] Finaliser le durcissement du endpoint REST public
- Remplacer la confiance implicite par des verifications strictes.
- Etat actuel re-verifie le 27 mars 2026:
- les projets inconnus sont deja rejetes
- un honeypot est deja present
- un rate limit par IP est deja present
- les tailles, nombres et types de fichiers sont deja controles
- les erreurs REST sont deja plus explicites qu'avant
- Reste a arbitrer si un durcissement supplementaire est necessaire:
- schemas encore plus stricts pour projectId / projectSlug / submissionId
- Ajouter un mecanisme anti-spam:
  token signe, nonce adapte au runtime, ou conservation du couple honeypot + rate limit actuel.

[x] Durcir le handler ZIP
- Refuser tout fichier executable ou interpretable:
  .php, .phtml, .phar, .sh, .cgi, etc.
- Verifier le contenu du ZIP avant extraction.
- N'accepter qu'une structure de package attendue.
- Imposer un manifeste.
- Imposer une liste blanche d'extensions autorisees.
- Verifier le slug final du package avant installation.
- S'assurer qu'aucun path traversal n'est possible.
- Point re-verifie le 27 mars 2026:
- ces controles sont maintenant presents dans le validateur ZIP.


Bloc B - Conformite et privacy

[x] Ajouter les integrations privacy WordPress
- Ajouter un personal data exporter.
- Ajouter un personal data eraser.
- Ajouter le texte suggere pour la privacy policy.
- Documenter clairement quelles donnees sont stockees et pourquoi.
- Point re-verifie le 27 mars 2026:
- exporter, eraser et texte privacy sont deja presents.

[x] Ajouter uninstall.php
- Nettoyer les options du plugin si desinstallation complete souhaitee.
- Definir clairement la politique de conservation des submissions et attachments.
- Point re-verifie le 27 mars 2026:
- uninstall.php existe deja, supprime les options plugin, et preserve desormais les submissions par defaut sauf suppression explicite.

[ ] Clarifier la gestion des pieces jointes
- Definir si les attachments doivent etre supprimes quand une submission est supprimee.
- Documenter le comportement dans le readme.
- Eviter les fichiers orphelins sans politique claire.
- Point re-verifie le 27 mars 2026:
- les attachments sont deja supprimes a la desinstallation
- le point restant est surtout la clarification de la politique de retention / suppression

Bloc D - Hygiene technique

[ ] Verifier l'internationalisation
- S'assurer que toutes les chaines visibles passent par les fonctions i18n.
- Verifier la coherence du text domain xpressui-bridge.

[ ] Verifier les sanitizations et escapings
- Sanitize de toutes les entrees admin et REST.
- Escape de toutes les sorties HTML, attributs, URLs et JSON.

[ ] Revoir les capabilities
- Confirmer que chaque ecran admin et action destructive ont la capability adequate.
- Verifier les actions de changement de statut et uploads.

[ ] Revoir les performances de l'admin
- Eviter les get_posts(posts_per_page => -1) si le volume peut croitre.
- Envisager des requetes plus ciblees pour les vues d'inbox et filtres.


Bloc E - Verification avant envoi

[ ] Lancer Plugin Check

[ ] Valider le readme

[ ] Tester sur une installation WordPress propre
- Activation
- Upload package
- Shortcode
- Soumission simple

- Notifications email
- Changement de statut

[ ] Tester sur la derniere version stable de WordPress
- Au 25 mars 2026, viser WordPress 6.9.x

[ ] Verifier l'absence de notices PHP et warnings JS
