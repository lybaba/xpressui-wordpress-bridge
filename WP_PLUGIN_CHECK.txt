WP Plugin Check - XPressUI WordPress Bridge
Date: 2026-03-25

Contexte

Analyse du plugin local situe dans:
dev/wordpress/xpressui-wordpress-bridge

Objectif:
Identifier les points a risque pour une soumission sur wordpress.org, avec priorite sur ce qui peut accelerer l'acceptation du plugin.


Constat principal

Dans l'etat, je ne miserais pas sur une acceptation rapide. Le plugin est propre sur plusieurs bases WordPress, mais il a 2 risques majeurs de revue, dont 1 potentiellement structurel.


Findings

1. Le plus gros risque est l'architecture "upload d'un ZIP executable puis iframe depuis uploads".

References:
- includes/admin-pages.php:315
- includes/shortcode.php:45

Le plugin extrait un package externe dans wp-content/uploads/xpressui/, puis sert son index.html dans un iframe. Pour l'equipe plugins, cela peut ressembler a du code executable charge hors du plugin lui-meme, ce qui touche directement la guideline 8, et possiblement la 6 si le service externe n'est pas cadre de facon tres nette.

Si l'objectif est de maximiser les chances d'acceptation rapide, le meilleur refactor serait:
- package declaratif JSON/config uniquement
- moteur de rendu inclus localement dans le plugin
- pas de HTML/JS "app" execute depuis uploads


2. Le endpoint REST public permet a n'importe qui de creer des soumissions privees et des pieces jointes.

Reference:
- includes/rest-endpoint.php:16

permission_callback => __return_true peut etre acceptable uniquement si tout le reste est fortement securise. Ici, il manque au minimum:
- validation stricte du projet cible
- rejet des slugs inconnus
- token de soumission signe par workflow
- limites de taille, nombre et type de fichiers
- anti-spam / rate limiting / honeypot
- sanitation plus stricte de projectId, projectSlug, submissionId

Tel quel, c'est un point qui peut facilement declencher un retour reviewer du type "security hardening required".


3. Le handler ZIP est trop permissif pour une soumission sereine.

Reference:
- includes/admin-pages.php:315

Le code verifie surtout l'extension, deplace le fichier, puis dezippe l'archive entiere. A renforcer avant soumission:
- manifeste obligatoire
- nom de dossier cible controle
- inspection du contenu avant extraction
- refus explicite de .php, .phar, .phtml, .sh, etc.
- liste blanche d'assets autorises
- verification que le package ne contient que des assets statiques attendus


4. Le plugin gere des donnees personnelles sans integration privacy WordPress.

References:
- includes/helpers.php:67
- includes/notifications.php:46

Le plugin stocke noms, email, telephone, payload complet et fichiers. Ce n'est pas forcement bloquant a lui seul, mais pour accelerer l'approbation il est recommande d'ajouter:
- exporter de donnees personnelles
- eraser de donnees personnelles
- texte suggere pour la politique de confidentialite
- strategie claire de suppression des attachments
- uninstall.php pour nettoyer options et CPT si souhaite


5. Le readme.txt est globalement bon, mais il faut le remettre au niveau avant depot.

Reference:
- readme.txt:1

Points positifs:
- 5 tags, donc conforme
- Stable tag = 1.0.0, coherent
- taille d'environ 6.2 KB, donc sous la zone a risque

Points a corriger:
- Tested up to: 6.7 est date. Au 25 mars 2026, la derniere release stable maintenue est 6.9.4, sortie le 11 mars 2026. Il est recommande de passer le readme a 6.9.
- le readme doit expliquer encore plus explicitement la dependance au service XPressUI, ses conditions d'usage et la privacy
- prevoir les assets WordPress.org (banner, icon, screenshot-1.png, etc.) avant envoi SVN


Ce qui est deja bien

- Prefixage coherent xpressui_*
- Nonces et checks de capabilities cote admin
- Echappement correct dans la majorite des sorties
- Pas d'obfuscation
- Header plugin/readme coherents
- UI admin sobre, sans pratiques "spammy"


Chemin le plus rapide vers une acceptation

1. Refactorer le modele de package: donnees/config seulement, pas de bundle HTML/JS executable uploade.
2. Durcir fortement le endpoint public et les uploads.
3. Ajouter le minimum privacy WordPress.
4. Mettre le readme a jour pour WordPress 6.9 et documenter tres clairement le service externe.
5. Passer Plugin Check + readme validator avant soumission.


Limites de cette verification

Je n'ai pas pu lancer php ni Plugin Check dans cet environnement, car php n'est pas installe ici. Cette note repose donc sur une revue statique du code.

Note importante sur le refactor structurel a prevoir

Le point sensible n'est pas seulement l'upload ZIP, mais surtout ce que le plugin fait apres extraction:
- aujourd'hui, le plugin extrait un mini-site dans uploads
- puis affiche son index.html dans une iframe

Pour wordpress.org, cela peut etre interprete comme du code applicatif externe execute en dehors du plugin.

Approche recommandee

Conserver le concept de "workflow package", mais transformer le package en contenu declaratif au lieu d'un bundle executable.
Le plugin doit devenir responsable du rendu.

Concretement:
1. definir un format de package autorise
- manifest.json
- fichiers de configuration JSON
- assets statiques eventuels
- pas de index.html applicatif
- pas de JS metier externe

2. stocker ces donnees dans uploads/xpressui/<slug>/
- mais seulement comme donnees et assets

3. remplacer l'iframe par un rendu natif cote plugin
- soit via shortcode PHP
- soit via un script frontend local au plugin
- le script lirait le manifest et construirait l'interface

4. faire poster le formulaire vers le endpoint REST du plugin

Pourquoi cette approche est preferable

- plus de mini-app executee depuis uploads
- logique metier visible dans le plugin
- plus simple a justifier a l'equipe wordpress.org
- meilleur controle sur la securite et la compatibilite

Version intermediaire possible

Si un refactor complet n'est pas possible tout de suite:
- interdire tout JS externe dans les packages
- interdire toute logique applicative dans le ZIP
- n'autoriser que HTML statique + assets
- deplacer progressivement le comportement dynamique dans un JS local au plugin

Cette version est meilleure que l'etat actuel, mais reste moins solide qu'un vrai modele declaratif.

Impact probable sur le code

- includes/admin-pages.php
  le validateur ZIP devrait a terme accepter un manifest.json comme point d'entree principal
- includes/shortcode.php
  l'iframe devrait etre remplacee par un conteneur rendu par le plugin
- nouveau fichier frontend JS local au plugin
  il lirait les donnees du manifest et construirait l'UI
- helpers supplementaires
  pour charger, valider et normaliser le manifest

Plan concret pour ce chantier plus tard

1. inspecter la structure reelle d'un package exporte XPressUI
2. definir le plus petit manifest exploitable
3. implementer un premier rendu natif simple
4. garder la soumission REST compatible
5. supprimer la dependance a index.html dans le shortcode

Precondition recommandee

Avant de coder ce refactor, il faut idealement analyser un vrai package exporte XPressUI afin de ne pas inventer un format a l'aveugle.


Strategie free / pro recommandee

Objectif

Rendre le plugin wordpress.org autonome, utile seul, et acceptable par l'equipe WordPress.org, tout en preparant une extension pro separee.

Version free recommandee

Le plugin wordpress.org doit permettre un usage complet sur un perimetre reduit mais utile.

A inclure dans le free:
- runtime xpressui-light embarque dans le plugin
- templates PHP locaux
- renderer backend base sur le JSON config
- 1 pack demo inclus et preinstalle
- shortcode fonctionnel sans compte externe obligatoire
- soumission REST
- stockage des submissions
- notifications email
- upload de fichiers basique
- sous-ensemble de champs bien choisis

Sous-ensemble free suggere:
- text
- email
- tel
- textarea
- select
- radio
- checkbox
- file upload simple
- sections / etapes simples
- theme basique

Le free doit etre un vrai produit utile, pas une simple demo marketing.

Version pro recommandee

Le pro doit etendre proprement le free, sans donner l'impression de "trialware".

A mettre en pro:
- types de champs avances
  - product-list
  - quiz
  - select-image
  - galleries complexes
  - widgets riches
- theming avance
  - styles premium
  - layouts avances
  - media/background avances
- logique avancee
  - conditional logic
  - regles metier complexes
  - scoring / branching sophistique
- integrations avancees
  - export
  - webhooks
  - CRM
  - analytics
- outils d'equipe avances
  - assignment avance
  - vues operateur enrichies
  - reporting

Ce qu'il faut eviter

Dans la version free, il faut eviter:
- une UI cassee pour les champs pro
- des fonctions incluses mais volontairement verrouillees
- une obligation de compte externe pour tester
- un telechargement dynamique de code premium

A faire a la place:
- le free ignore proprement les composants non supportes
- ou affiche un message admin clair si un workflow utilise des composants pro
- et cote frontend, un fallback propre si le workflow depasse les capacites du free

Regle produit simple

Le free doit pouvoir tenir cette promesse:
"Creer, embarquer et recevoir des soumissions pour des formulaires WordPress utiles."

Le pro peut porter la promesse:
"Construire des experiences avancees, visuelles et metier."

Structure technique recommandee

- xpressui-wordpress-bridge
  - runtime light
  - renderer PHP/JS local
  - pack demo inclus
  - validateur de workflow free-compatible
- xpressui-wordpress-bridge-pro
  - surcharge/extension du renderer
  - registre de composants pro
  - styles et comportements avances

Compatibilite des workflows

Il est recommande d'ajouter un manifeste avec capacites declarees, par exemple:
- requires_runtime: light | pro
- features: [...]
- components: [...]

Ainsi, le plugin free peut:
- accepter les workflows light
- refuser ou degrader proprement les workflows pro


Protection de l'extension pro

Constat

Si le code PHP/JS du plugin pro est livre au client, il n'existe pas de protection parfaite. On peut compliquer l'usage non autorise, pas le rendre impossible.

La bonne strategie repose sur 3 niveaux.

1. Protection produit

Le plugin pro doit verifier une licence valide avant d'activer les fonctions premium.

Mecanisme recommande:
- ecran d'activation licence
- cle liee a un compte
- validation aupres de l'API IAKPress
- stockage local d'un jeton signe avec date d'expiration
- revalidation periodique, pas a chaque requete

Important:
- ne pas casser brutalement un site en production si la licence expire
- mieux vaut bloquer les nouvelles activations/fonctions pro que rendre le front inutilisable

2. Protection architecture

Il faut eviter de mettre toute la valeur dans du code purement local.

Les capacites premium les plus robustes sont celles qui dependent d'un service controle par l'editeur, par exemple:
- generation de packs pro
- compilation avancee de workflow
- composants premium exportes seulement pour licences valides
- synchronisation ou services metier cote API

Cette approche est plus robuste qu'un simple "if licence valide" en PHP.

3. Protection commerciale

La licence doit offrir une valeur continue superieure au cout de la fraude.

Valeurs a associer a l'achat:
- mises a jour
- support
- templates premium
- nouveaux composants
- acces au builder/export pro
- compatibilite garantie

Modele technique recommande pour xpressui-wordpress-bridge-pro

- plugin separe
- verification que le free est actif
- stockage d'une licence locale signee
- si licence absente ou invalide:
  - ne pas enregistrer les renderers/composants pro
  - afficher des notices admin
  - laisser le site stable
- les workflows marques requires_runtime: pro ne doivent pas s'executer sans licence valide

Ce qu'il faut eviter pour la protection

- obfuscation agressive du PHP
- ionCube ou mecanismes similaires
- appels API obligatoires a chaque page vue
- desactivation brutale du site si la licence n'est plus valide

Le modele le plus propre

- le free sait lire un manifest
- le pro enregistre des capacites supplementaires
- la licence controle l'activation de ces capacites
- les exports/workflows pro sont idealement generes depuis le backend editeur uniquement pour les clients licencies


Ordre de priorite recommande maintenant

Le chantier le plus rentable a court terme est:
1. definir le manifeste de compatibilite des workflows

Pourquoi:
- il servira de base au free/pro
- il servira de base au validateur de package
- il servira de base au futur refactor du rendu
- il permettra de savoir clairement ce que le free accepte ou refuse

Ensuite, ordre recommande:
2. definir le perimetre exact du runtime light
3. ajouter un pack demo embarque
4. commencer le renderer WordPress natif a partir du JSON config
5. preparer ensuite l'extension pro et la logique de licence

En synthese:
- le plus rentable maintenant = concevoir le manifest et les capabilities
- le plus sensible ensuite = rendre le plugin free autonome
- la licence pro vient apres, quand les frontieres free/pro sont clairement definies


Mise a jour du suivi - 25 mars 2026

- pipeline Jinja -> PHP en place pour les templates WordPress embarques
- le plugin free n'embarque plus que les templates compatibles avec le runtime light
- les packs WordPress exportent maintenant `template.context.json` en plus du manifest et de `form.config.json`
- le shell WordPress peut rendre un template PHP compile avec fallback de securite
- les templates systeme du shell utilisent maintenant la traduction WordPress pour les chaines bridge
- le plugin charge son textdomain `xpressui-bridge`
- generation du fichier POT automatisee et verifiee par test
- les messages systeme PHP et JS du shell utilisent maintenant une charge utile i18n commune
